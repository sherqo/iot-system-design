Step 1: Purpose & Requirements 

    Purpose of the IoT System 

The purpose of the Smart Mosque Management System is to design an intelligent IoT-based solution that enhances the operational efficiency of mosques by: 

    Optimizing energy consumption (lighting and air conditioning)  

    Conserving water usage in wudu (ablution) areas  

    Monitoring and managing crowd occupancy  

    Improving indoor air quality and comfort  

The system leverages real-time sensor data, automated control mechanisms, and a centralized monitoring dashboard to ensure efficient resource utilization, especially during high-attendance periods such as Friday prayers and Ramadan. 

    Functional Requirements 

These define what the system must do: 

A. Energy Management 

    Automatically turn ON/OFF lights and fans/AC based on:  

    Prayer times (via API)  

    Occupancy detection (motion sensors)  

    Adjust energy usage dynamically depending on crowd density  

B. Occupancy Detection 

    Count number of people entering/exiting using IR sensors  

    Provide real-time hall capacity status  

    Trigger alerts if capacity exceeds safe limits  

C. Water Monitoring (Wudu Area) 

    Measure water flow using flow sensors  

    Detect abnormal usage (leaks or taps left open)  

    Automatically close water supply using water valves  

D. Air Quality Monitoring 

    Measure air quality (e.g., CO₂, humidity, temperature)  

    Trigger ventilation (fans) when thresholds are exceeded  

E. Central Dashboard 

    Display:  

    Current occupancy  

    Energy consumption  

    Water usage  

    Air quality readings  

    Allow manual override control for administrators  

F. Notifications & Alerts 

    Send alerts for:  

    Overcrowding  

    Water leakage  

    Poor air quality  

    System faults  

    Non-Functional Requirements 

These define how well the system should perform: 

A. Performance 

    Real-time response (low latency in sensor-actuator loop)  

    Efficient handling of multiple sensors simultaneously  

B. Scalability 

    System should support small to large mosques  

    Easy addition of new sensors/devices  

C. Reliability 

    Continuous operation with minimal downtime  

    Fault-tolerant communication between devices  

D. Security 

    Encrypted Communication: Use WPA2-PSK encryption for all wireless sensor-to-gateway connections. 

    Access Control: Secure the monitoring dashboard with username/password authentication to prevent unauthorized overrides. 

    Device Filtering: Use MAC address filtering to ensure only authorized IoT nodes can join the network. 

E. Usability 

    Simple and intuitive dashboard interface  

    Minimal training required for mosque staff  

F. Maintainability 

    Easy to update system (software + hardware)  

    Modular design for replacing faulty components  

G. Cost Efficiency 

    Use affordable IoT components suitable for local deployment in Egypt 

    Constraints and Assumptions 

Constraints 

    Limited budget → use cost-effective sensors  

    Implementation in Cisco Packet Tracer simulation  

    Dependence on internet for prayer-time API  

Assumptions 

    Stable local network exists in mosque  

    Power supply is reliable  

    Admin has basic technical knowledge 

The system aligns with Egypt’s sustainability goals by reducing unnecessary consumption of water and electricity in high-usage public religious spaces. 

Step 2: Process Specification 

2.1 Overview 

The Smart Mosque Management System operates through a set of coordinated processes that continuously collect sensor data, analyze conditions, and trigger automated actions. These processes ensure efficient energy usage, water conservation, and safe occupancy management. 

The system follows a Sense → Process → Act → Monitor pipeline. 

2.2 Core System Processes 

2.2.1 Occupancy Detection Process 

Objective: Monitor the number of people inside the mosque. 

Workflow: 

    IR sensors at entrances detect entry/exit events  

    System increments/decrements occupancy count  

    Data is sent to the central controller  

    If occupancy exceeds threshold:  

    Trigger alert on dashboard  

    Occupancy data is used by other processes (energy control) 

2.2.2 Energy Management Process 

Objective: Optimize electricity usage based on prayer times and occupancy. 

Workflow: 

    System retrieves prayer times from API  

    Motion sensors detect presence in prayer hall  

    Decision logic:  

    IF (Within Prayer Window OR Motion Detected)  
    → Turn ON lights and AC/fans  

    ELSE 
    → Turn OFF devices  

    Adjust intensity (optional) based on crowd density  

2.2.3 Wudu Water Monitoring Process 

Objective: Prevent water waste in ablution areas. 

Workflow: 

    Flow sensors measure water usage  

    System checks:  

    Continuous flow without interruption  

    If abnormal usage detected:  

    Assume tap left open / leak  

    Activate water valve to shut water  

    Log water consumption data  

2.2.4 Air Quality Monitoring Process 

Objective: Maintain comfortable and safe air condisations. 

Workflow: 

    Sensors read CO₂, humidity, temperature  

    Compare readings with predefined thresholds  

    IF air quality is poor:  

    Activate ventilation (fans/windows system)  

    Update dashboard with real-time values  

2.2.5 Central Monitoring & Control Process 

Objective: Provide full system visibility and control. 

Workflow: 

    Collect data from all subsystems  

    Display on web dashboard:  

    Occupancy  

    Energy usage  

    Water consumption  

    Air quality  

    Allow admin to:  

    Override automation  

    Turn devices ON/OFF manually  

    Store historical data (optional)

2.4 Process Interaction Notes  

    Occupancy data → affects energy decisions  

    Air quality → may override energy logic (force ventilation ON)  

    Water system → independent but logged centrally  

    All processes communicate via a central controller (MCU/SBC) 

The system processes are designed as loosely coupled modules to ensure scalability, maintainability, and real-time responsiveness

5.3 Service Descriptions 

 

SVC-01 — Prayer Time Sync Service 

    Purpose: Retrieves accurate daily prayer times to anchor all energy automation logic. Without this, the system cannot distinguish active prayer windows from idle periods. 

    Interface: HTTP GET request to a prayer time API (e.g., aladhan.com/v1/timingsByCity?city=Cairo&country=Egypt) once per day at midnight. 

    Output: Five daily timestamps (Fajr, Dhuhr, Asr, Maghrib, Isha) stored in the ESP32's local memory. 

    Fallback: If the API is unreachable, the system uses the previous day's cached schedule. The drift between consecutive days for Egyptian cities is under 2 minutes — acceptable for this use case. 

    Data produced:

  "asr": "15:28", 
  "maghrib": "18:44", 
  "isha": "20:10"

SVC-02 — Occupancy Tracking Service 

    Purpose: Counts the number of people inside the prayer hall in real time to support energy decisions and crowd management. 

    Interface: PIR sensor events at entry and exit points → ESP32 counter logic → count exposed via local API endpoint polled by the dashboard every 30 seconds.

Step 6: IoT Level Specification 

The Smart Mosque Management System is structured across five IoT levels, following the standard layered IoT architecture. Each level has a defined responsibility, and layers communicate only with adjacent layers to maintain separation of concerns. 

Level 1 — Perception Layer 

Role: Direct interaction with the physical environment. All sensing and actuation hardware lives here.

Communication from this layer: Digital GPIO signals and I²C/UART data lines to the ESP32 processing unit at Level 2. 

 

Level 2 — Network Layer 

Role: Local connectivity between sensors/actuators and the processing unit. Handles data transport within the mosque premises. 

    Hardware: Local Wi-Fi access point connecting ESP32 units and the local server. 

    Protocol: Wi-Fi (IEEE 802.11), secured with WPA2-PSK encryption. 

    Topology: Star — all ESP32 nodes connect to the central access point. 

    Segmentation: IoT devices are on a separate network segment from general user devices (guest Wi-Fi), reducing interference and unauthorized access risk. 

    Real-world equivalent: In a production deployment, Zigbee or MQTT over Wi-Fi to a local broker would be used. In the Cisco Packet Tracer simulation, this is represented by the local LAN connecting MCU units to the Registration Server. 

Key design decision: The core control loop (sensor → decision → actuator) is handled locally at Level 3 and does not depend on internet connectivity. The network layer only carries data; it does not make decisions. 

 

Level 3 — Edge Processing Layer 

Role: All decision-making and real-time control logic. This is the brain of the system. 

    Hardware: ESP32 microcontroller units — one per functional zone (prayer hall zone, wudu area zone). 

    Why ESP32: Built-in Wi-Fi, multiple GPIO pins, I²C/UART/SPI support, low power modes, and sufficient processing capability for concurrent sensor polling and actuator control. Suitable for continuous operation in a mosque environment.

Failure behavior: If a sensor goes offline, the block flags it to the Processing Block. The system then applies a safe default: assumes occupied (for energy decisions) and assumes flow present (for water decisions) until the sensor is restored. 

 

Block 2 — Processing Block 

The decision-making core. All logic that determines what actions to take lives here, running on the ESP32. 

Sub-functions: 

Occupancy Counter 

    Maintains a running count of people in the hall. 

    Enforces a floor of 0 — exit events when count is already 0 are ignored. 

Prayer Window Evaluator 

    Compares current system time to the cached prayer schedule from SVC-01. 

    Returns a boolean in_prayer_window used by the Decision Engine. 

Decision Engine — Energy 

IF in_prayer_window AND occupancy > 0: 
    → ACTIVATE lights and fans for occupied zones 
ELSE: 
    → DEACTIVATE lights and fans 
IF override_active: 
    → Apply admin override command regardless of above 

Decision Engine — Air Quality 

IF co2 > 1000 ppm OR temperature > 28°C OR humidity > 65%: 
    → ACTIVATE fans (independent of prayer window logic) 
ELSE: 
    → Return fans to energy control logic 

Decision Engine — Water 

IF flow_rate > 0 AND no_motion_duration > 30s: 
    → CLOSE solenoid valve 
    → Trigger ALT-03 
IF flow_rate > leak_threshold AND no_motion: 
    → CLOSE solenoid valve 
    → Trigger ALT-04 (higher priority) 
IF flow_rate == 0: 
    → No action 

State Manager 

    Maintains current state of every device, sensor reading, and counter. 

    Acts as the single source of truth at the edge. 

    Pushes state snapshots to the Data & Storage Block every 30 seconds. 

 

Block 3 — Actuation Block 

Executes commands from the Processing Block on physical devices.

Feedback loop: After every actuation command, the block waits up to 3 seconds for a confirmation signal. If no confirmation is received after 3 retries, the Processing Block logs a device fault and triggers ALT-07 in the Alert Block. This ensures the system knows whether commands are actually being executed, not just sent.

Block 5 — Alert Block 

Monitors threshold conditions and pushes notifications to the Dashboard Block. 

    Subscribes to state change events published by the Processing Block. 

    Evaluates all seven alert conditions defined in SVC-06. 

    Deduplicates: same alert type for same device/zone fires at most once per 5 minutes. 

    Stores each alert with timestamp, zone, severity, and resolution status. 

    Pushes active alerts to the dashboard in real time. 

 

Block 6 — Dashboard Block 

The admin's interface for monitoring and control. 

    Polls the Data & Storage Block every 30 seconds for current readings. 

    Renders four monitoring panels: occupancy, energy, water, air quality. 

    Displays the active alert queue and prayer schedule. 

    Accepts manual override inputs from the admin and forwards them as commands to the Processing Block. 

    Single admin role — no public access. 

 

Block 7 — External API Block 

Interfaces with the only external dependency: the prayer time service. 

    Sends one HTTP GET request per day at midnight. 

    Parses the five daily prayer timestamps and writes them to ESP32 local memory. 

    If the request fails, uses the previous day's cached schedule as fallback. 

    No other external API dependencies — all other logic is self-contained on the local network.
